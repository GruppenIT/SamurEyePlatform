# Phase 5 — UI Design Contract

> Visual and interaction contract for Phase 5: EDR Timestamps.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase UI Boundary

**This phase has no new UI surfaces.**

Per CONTEXT.md (Phase Boundary): "No new scanner capabilities, no UI changes, no new journey types."

All work is confined to `shared/schema.ts`, `server/services/scanners/edrAvScanner.ts`,
`server/storage/database-init.ts`, and a new storage helper. The two new timestamp fields
(`deploymentTimestamp`, `detectionTimestamp`) will appear in existing API response payloads
consumed by existing UI components — no component changes are required to display them.

The design contract below documents the **existing established system** that governs any
incidental rendering changes (e.g., if an existing component already maps EdrFinding fields
to display, the new optional fields will appear as undefined/absent and must not break layout).

## Typography

| Role | Size | Weight | Line Height | Font | Source |
|------|------|--------|-------------|------|--------|
| Body | 14px | 400 (regular) | 1.5 | Inter | existing codebase |
| Label | 12px | 400 (regular) | 1.4 | Inter | existing codebase |
| Heading | 20px | 600 (semibold) | 1.2 | Inter | existing codebase |
| Display | 28px | 600 (semibold) | 1.2 | Inter | existing codebase |

**Declared weights: 2 — 400 (regular) and 600 (semibold).** Label (12px) is differentiated
from Body (14px) by size contrast alone; no separate medium weight is required.

**Timestamp field rendering rule:** `deploymentTimestamp` and `detectionTimestamp` values,
when surfaced in any UI component, MUST be rendered in `font-mono` (JetBrains Mono) at the
existing body size (14px, weight 400) to distinguish machine-generated ISO-8601 strings from
human-readable text. No separate type-scale entry is introduced — the mono face is applied
via the `font-mono` utility class at the standard 14px body size.

## Color

All values are CSS custom properties defined in `client/src/index.css`.

| Role | CSS Variable | Resolved Value | Usage |
|------|-------------|----------------|-------|
| Dominant (60%) | `--background` | hsl(222, 18%, 8%) | Page background, main surface |
| Secondary (30%) | `--card` / `--sidebar` | hsl(222, 16%, 13%) / hsl(222, 18%, 10%) | Cards, sidebar, nav |
| Accent (10%) | `--primary` | hsl(197, 75%, 48%) | Reserved for: active sidebar item border, primary CTA buttons, ring/focus indicators, active job shimmer animation, status-mitigated indicator |
| Accent 2 | `--accent` | hsl(45, 80%, 60%) | Reserved for: medium severity indicators, status-investigating indicator |
| Destructive | `--destructive` | hsl(0, 75%, 55%) | Destructive actions, critical severity, status-open, status-error indicators |

**Semantic severity tokens** (read-only — do not redefine in this phase):

| Token | Value | Usage |
|-------|-------|-------|
| `--severity-critical` | hsl(0, 75%, 55%) | Critical threats |
| `--severity-high` | hsl(25, 85%, 50%) | High threats |
| `--severity-medium` | hsl(45, 80%, 55%) | Medium threats |
| `--severity-low` | hsl(142, 60%, 40%) | Low threats / detected (EDR success) |
| `--severity-info` | hsl(210, 50%, 55%) | Informational |

**EDR detection semantic mapping** (if `detectionTimestamp` or `detected` boolean surfaces
in future UI — follow this convention):
- `detected: true` → `--severity-low` (green) — EDR caught the threat
- `detected: false` → `--severity-critical` (red) — EDR missed the threat
- `detected: null` → `--muted-foreground` — no result / not tested

Source: existing `client/src/index.css`.

## Copywriting Contract

This phase introduces no new UI-visible states. The contract below covers existing states
that the executor must NOT modify, plus one new data-surface note.

| Element | Copy | Status |
|---------|------|--------|
| Primary CTA | Not applicable — no new interactive elements in this phase | N/A |
| Empty state heading | Not applicable — no new empty states introduced | N/A |
| Empty state body | Not applicable | N/A |
| Error state | Not applicable — DB insert errors are server-side only (logged, never surfaced to UI) | N/A |
| Destructive confirmation | Not applicable — no destructive UI actions in this phase | N/A |

**Data surface note:** The new `deploymentTimestamp` and `detectionTimestamp` fields join the
existing EdrFinding payload. If any existing component renders these fields:
- Null value display: render as `—` (em dash), not "null", "undefined", or empty string.
- Timestamp format: display as locale date-time string (`toLocaleString()`), not raw ISO-8601,
  unless displayed in a technical/debug context where `font-mono` is appropriate.
