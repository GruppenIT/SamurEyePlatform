# Milestone v1.0: SamurEye Product Revision

**Status:** SHIPPED 2026-03-17
**Phases:** 1-4
**Total Plans:** 12

## Overview

Deep revision of the SamurEye platform scan pipeline in 4 phases following strict data dependency order. Parsers stabilized before threat engine grouping/scoring. Scoring stabilized before remediation templates. Remediation records created before UI could display actionable plans. Each phase completed a coherent capability layer and unblocked the next.

## Phases

### Phase 1: Parser Foundation

**Goal**: All 4 scanner parsers produce rich, typed, validated output that the threat engine can consume without data loss
**Depends on**: Nothing (first phase)
**Requirements**: PARS-01, PARS-02, PARS-03, PARS-04, PARS-05, PARS-06, PARS-07, PARS-08, PARS-09, PARS-10, PARS-11
**Plans**: 3 plans

Plans:
- [x] 01-01: nmap parser rewrite — switch invocation to `-oX -` XML output and parse with fast-xml-parser, capturing OS detection, service version details (PARS-01, PARS-03, PARS-04)
- [x] 01-02: nmap NSE and nuclei parsers — capture full NSE script blocks with CVE refs; validate nuclei JSONL line-by-line with Zod, capturing matcher-name, extracted-results, curl-command (PARS-02, PARS-05, PARS-06)
- [x] 01-03: AD and EDR parsers + typed interfaces — add `-Depth 10` to PowerShell scripts, capture full AD structures, produce per-host EDR timeline, define NormalizedFinding interfaces and Zod schemas for all 4 parsers, write snapshot tests (PARS-07, PARS-08, PARS-09, PARS-10, PARS-11)

### Phase 2: Threat Engine Intelligence

**Goal**: Related findings are consolidated into grouped threat clusters and every threat carries a contextual severity score stored at persistence time
**Depends on**: Phase 1
**Requirements**: THRT-01, THRT-02, THRT-03, THRT-04, THRT-05, THRT-06, THRT-07, THRT-08, THRT-09, THRT-10
**Plans**: 3 plans

Plans:
- [x] 02-01: Schema migrations — add contextual_score, score_breakdown, projected_score_after_fix columns; create posture_snapshots and recommendations tables; add indexes
- [x] 02-02: Threat grouping engine — groupFindings() with journey-specific grouping keys; preserve existing correlation key format
- [x] 02-03: Contextual scoring engine — ScoringEngineService with weighted formula; persist score_breakdown as JSONB; write posture_snapshots on journey completion

### Phase 3: Remediation Engine

**Goal**: Every threat group has a specific, actionable remediation that references actual hosts, ports, and service versions found
**Depends on**: Phase 2
**Requirements**: REMD-01, REMD-02, REMD-03, REMD-04, REMD-05, REMD-06, REMD-07
**Plans**: 2 plans

Plans:
- [x] 03-01: Recommendation engine core — 25 template functions, recommendationEngine singleton, storage CRUD, pipeline integration after scoring
- [x] 03-02: Remediation lifecycle — sync recommendation status on threat status changes, recommendation API endpoints

**Depends on**: Phase 3
**Requirements**: UIFN-01, UIFN-02, UIFN-03, UIFN-04, UIAP-01, UIAP-02, UIAP-03, UIAP-04, UIDB-01, UIDB-02, UIDB-03, UIDB-04, UIDB-05, UIDB-06

## Milestone Summary

**Key Decisions:**
- Contextual templates over AI/LLM for recommendations — static template functions per threat type (validated: sufficient for v1)
- Improve existing parsers rather than rewrite from scratch — preserved working functionality
- Additive schema changes only — protected existing data with rollback capability
- Threat grouping at engine level — single source of truth for threat count/severity
- Self-referential parentThreatId with lambda references() for Drizzle circular init
- Static TypeScript template functions per rule ID over Handlebars for type safety
- Coverage endpoint uses 2 queries per journey type instead of complex JOIN for clarity
- GET /api/action-plan filters only open parent threats to avoid duplicate child actions

**Issues Resolved:**
- ESM type re-export issue on Windows (IStorage import/export type)
- Windows reusePort not supported (removed from server listen)
- getThreatsWithHosts missing parentThreatId/groupingKey/contextualScore fields
- Sparkline contrast too low in dark mode (switched to explicit blue-500 color)

**Issues Deferred:**
- edrAvScanner.test.ts has 7 pre-existing failures (missing C:\tmp\ directory)

**Known Gaps (7 requirements):**
- PARS-07: AD PowerShell -Depth 10 (parser depth improvement)
- PARS-08: AD full group membership chains
- PARS-09: EDR per-host timeline
- PARS-11: Snapshot tests for all 30+ rules (partial coverage)
- THRT-06: Scoring weight calibration (40/25/20/15 exact split)
- THRT-08: Host type criticality multiplier calibration
- THRT-09: Exploitability multiplier calibration

**Technical Debt Incurred:**
- Scoring weights are functional but not calibrated against real scan data distributions
- AD/EDR parsers improved but not fully capturing nested structures at maximum depth
- Snapshot test coverage is partial (key rules covered, not all 30+)
