# Phase 6 — UI Design Contract

> Visual and interaction contract for Phase 6: Calibration and Quality.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase UI Boundary

**This phase has no new UI surfaces.**

Per CONTEXT.md (Phase Boundary): "No new scoring capabilities, no new threat rules, no UI changes."

All work is confined to:
- `scripts/calibrate.ts` — CLI calibration script (Node.js, no browser rendering)
- `server/__tests__/scoringEngine.test.ts` — calibration regression tests (Vitest, no browser rendering)
- `server/__tests__/edrAvScanner.test.ts` — verify zero failures (no changes expected)
- `server/__tests__/threatRuleSnapshots.test.ts` / `.snap` — snapshot completeness check
- `.planning/phases/06-calibration-and-quality/CALIBRATION-REPORT.md` — plaintext report

No browser-rendered components, routes, modals, or pages are introduced or modified.
The calibration script outputs to stdout and writes a markdown file; neither surface is
governed by the UI design system.

The design contract below documents the **existing established system** that governs any
incidental rendering concerns (e.g., if scoring weight constants change in `scoringEngine.ts`,
existing UI components that display threat scores must continue to render without layout breaks).

**Declared weights: 2 — 400 (regular) and 600 (semibold).**

**Scoring value rendering rule:** Numeric threat scores produced by `scoringEngine.ts` and
displayed anywhere in existing UI MUST be rendered as `font-mono` (JetBrains Mono) at body
size (14px, weight 400). Score breakdowns (the 7-field object) must not wrap onto multiple
lines in table cells — use `tabular-nums` and `whitespace-nowrap` where scores appear inline.
No new type-scale entries are introduced.

## Color

All values are CSS custom properties defined in `client/src/index.css`.

| Role | CSS Variable | Resolved Value | Usage |
|------|-------------|----------------|-------|
| Dominant (60%) | `--background` | hsl(222, 18%, 8%) | Page background, main surface |

| Accent (10%) | `--primary` | hsl(197, 75%, 48%) | Reserved for: active sidebar item border, primary CTA buttons, ring/focus indicators, active job shimmer animation, status-mitigated indicator |
| Accent 2 | `--accent` | hsl(45, 80%, 60%) | Reserved for: medium severity indicators, status-investigating indicator |
| Destructive | `--destructive` | hsl(0, 75%, 55%) | Destructive actions, critical severity, status-open, status-error indicators |

**Semantic severity tokens** (read-only — do not redefine in this phase):

| Token | Value | Usage |
|-------|-------|-------|
| `--severity-critical` | hsl(0, 75%, 55%) | Critical threats |
| `--severity-high` | hsl(25, 85%, 50%) | High threats |
| `--severity-medium` | hsl(45, 80%, 55%) | Medium threats |
| `--severity-low` | hsl(142, 60%, 40%) | Low threats |
| `--severity-info` | hsl(210, 50%, 55%) | Informational |

**Scoring calibration impact note:** If `scoringEngine.ts` constants are patched by the
calibration script (THRT-06, THRT-08, THRT-09), the semantic severity color mapping must
remain unchanged. Severity level labels (critical / high / medium / low) are determined by
score thresholds defined in `scoringEngine.ts` — if those thresholds change, the CSS token
assignments above must remain stable (tokens map to labels, not raw score numbers).

Source: existing `client/src/index.css`.

This phase introduces no new UI-visible states. The contract below covers the one new
output artifact (CALIBRATION-REPORT.md) and the non-UI script stdout surface.

| Element | Copy | Status |

**Calibration report copy contract** (CALIBRATION-REPORT.md — plaintext, not browser UI):

| Section | Format |
|---------|--------|
| Report header | `# Calibration Report — {ISO date}` |
| Per-component result | `PASS` or `FAIL: {inversion description}` |
| Changes made | `CHANGED: {constant name} {old value} → {new value}` |
| No changes | `NO CHANGES — weights within tolerance` |
| Script stdout prefix | `[calibrate] {message}` — all stdout lines prefixed for grep-ability |

**Scoring display invariant:** Existing UI components that render threat scores must continue
to display numeric values with at most 1 decimal place (e.g., `7.4`, not `7.38291`). This
constraint already exists in the codebase; the calibration script must not introduce raw
floating-point values into scores stored in the database.

## Interaction Contracts

No new interactions. No new UI states. No new routes, pages, or modals.

The only downstream UI change is that existing threat score displays may show different
numeric values after calibration auto-patch, and some findings may receive a different
severity classification. These are data changes, not interaction changes — no component
code is modified.

**Zero-regression rule:** After Phase 6 completes, `npx vitest run` must exit 0. Any
snapshot update triggered by scoring constant changes must be committed before milestone
close (QUAL-02). The existing browser UI must render without console errors after any
`scoringEngine.ts` constant patch.
